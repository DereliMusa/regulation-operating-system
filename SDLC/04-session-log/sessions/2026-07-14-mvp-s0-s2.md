## Real problems found and fixed (not just "it worked")

1. **`vue-router` version mismatch.** S0 pinned `vue-router: ^4.5.0` out of habit; Nuxt
   4.4.8 actually depends on `vue-router@5.1.0`. The stale pin caused two `vue-router`
   copies to install, and `vue-tsc` resolved the wrong one, breaking `npm run typecheck`
   with `ERR_PACKAGE_PATH_NOT_EXPORTED` (missing `./volar/sfc-route-blocks` export). Fixed
   by bumping the pin; documented in `01-architecture/tech-stack.md`.
2. **`app.config.ts` silently ignored.** Placed at the repo root in S0; Nuxt resolves
   `app.config.ts` from the srcDir (`app/` in Nuxt 4), so it never applied and NuxtUI's
   primary color quietly fell back to its own default (green) instead of the brand blue.
   No error was thrown — this was only caught by visually screenshotting the login page and
   noticing the button wasn't blue, then confirming via `getComputedStyle`. Fixed by moving
   the file to `app/app.config.ts`; documented in `01-architecture/architecture.md`.
3. **DB singleton side effect broke test isolation.** `db.ts` originally created its
   file-based singleton at module scope; any test importing `createDb` from that file also
   triggered the singleton, opening the real database file — under Vitest's parallel workers
   this caused intermittent `SqliteError: database is locked`. Fixed by extracting the pure
   factory into `createDb.ts` (no module-level side effects); `db.ts` now only builds the
   app singleton from it. Documented in `02-standards/testing-strategy.md` as a pattern to
   repeat for any future factory+singleton module.
4. **Migrations path resolution.** `import.meta.url`-relative path resolution failed under
   Nitro's dev bundler (it doesn't preserve real source file locations), causing "Can't find
   meta/_journal.json file" at dev-server startup. Fixed by resolving via
   `process.cwd()` instead; flagged an S9 Dockerfile requirement (must `COPY
   server/database/migrations` into the image) in `01-architecture/deployment.md`.
